# Introduction

This article documents the analysis of a phishing infrastructure using the `amendes-justice-gov.xxx` naming pattern. The objective was not only to collect indicators, but also to understand the campaign logic, the operator workflow, and the technical design choices behind the phishing kit.

The methodology followed a simple sequence:

1. enumerate exposed routes and domains;
2. identify which hosts were really active;
3. group infrastructure by DNS and hosting patterns;
4. inspect browser traffic;
5. reverse the client-side logic;
6. understand how data was collected and transmitted.

This sequence matters because every step answers a different question. Enumeration tells us where the campaign lives. Traffic analysis tells us how it works. Reversing tells us what the operator is actually collecting.

# Enumerations

### Endpoints (/--tlds--)

The first thing we wanted to understand was whether this phishing page was isolated to one route or whether the site exposed several country- or workflow-specific endpoints.

```bash
ffuf -w tlds.txt -u https://amendes-justice-gov.beauty/FUZZ/ -fs 0 -mc 200,204,301,302,307,401,403
```

The logic here was simple: if the operator used a shared template for multiple targets, different paths might reveal country codes, internal routes, or alternate workflow stages.

Using `ffuf` on the path let me test many possible route names quickly. The interesting result was that `/ma/` appeared to be the meaningful route.

That matters because `/ma/` is a strong geographical clue. It suggests the workflow was built for Morocco, not just as a random fake payment page. In phishing investigations, even a small routing detail can tell you who the lure is trying to reach.

### TLD Enumeration

Once `/ma/` was identified as the interesting path, the next question became: is this only one domain, or part of a broader cluster?

```bash
ffuf -w tlds.txt -u https://amendes-justice-gov.FUZZ/ma/ -fs 0 -mc 200,204,301,302,307,401,403
```

Instead of fuzzing paths, we fuzzed the TLD itself. The reasoning was that phishing operators often register many variants of the same name across cheap or disposable TLDs. This gives them resilience:

- one domain can be burned while others stay alive;
- they can rotate links when reports increase;
- they can keep the same lure structure while moving traffic across domains.

This step was useful because it transformed the investigation from `one suspicious website` into `one phishing infrastructure pattern deployed across many domains.`

### DNS Enumeration and Infrastructure Grouping

Once the active domains were identified, the next step was to see whether they pointed to the same backend infrastructure.

```bash
#!/bin/bash

if [[ $# -ne 2 ]]; then
  echo "Usage: $0 <tlds_file> <base_domain>"
  exit 1
fi

tlds_file="$1"
base_domain="$2"

if [[ ! -f "$tlds_file" ]]; then
  echo "Error: File '$tlds_file' not found."
  exit 1
fi

for tld in $(cat "$tlds_file"); do
  domain="${base_domain}.${tld}"

  A=$(dig +short "$domain" A)
  CNAME=$(dig +short "$domain" CNAME)

  if [[ -n "$A" || -n "$CNAME" ]]; then
    echo "=== $domain ==="

    if [[ -n "$A" ]]; then
      echo "A record(s):"
      echo "$A"
    fi

    if [[ -n "$CNAME" ]]; then
      echo "CNAME:"
      echo "$CNAME"
    fi

    echo ""
  fi
done
```

TLD enumeration shows which domains are alive. DNS review shows whether those domains belong to the same infrastructure family.

That distinction matters because a phishing brand can attract unrelated lookalikes. If several domains resolve into the same IP ranges, same ASN, or same hosting patterns, confidence increases that they are part of the same operation.

This step also helped eliminate noise. Some domains resolved, but only to parking infrastructure, which created false positives. Cleaning those out was necessary before making claims about the campaign.

### Understanding the Encryption Logic

At first glance, the `42["message", ...]` payloads were unreadable. The next step was to determine whether they were just encoded, compressed, or actually encrypted.

we tested this by changing field values and comparing the outputs.

```txt title="name : amendes"
42["message","9J/UwM0nsmrdXqUVWny6zjo8nD559AWv58L4xtEANxIKG2hkdWpjUxPFT83e6Zqt8xNOWjvoA/LqUFDiW0nMBQ=="]
```

```txt title="name : amendesamendesamendes"
42["message","9J/UwM0nsmrdXqUVWny6zjo8nD559AWv58L4xtEANxIKG2hkdWpjUxPFT83e6ZqtBbd4HbauOnKxhMxlwdZSS6zDV77VCAA7pJSwf7XI68g="]
```

The reasoning was based on pattern observation.

When different plaintext values produce ciphertexts with a similar prefix and divergence later, that usually indicates block encryption behavior. It is not something you would expect from plain Base64 encoding or a simple obfuscation routine.

This was the clue that pushed the analysis toward JavaScript reversing instead of just traffic observation.

# JavaScript Reversing Logic

The next goal was to find where the client prepared and encrypted the data.

By placing breakpoints around keyboard input handlers and the functions triggered on field changes, we could follow the path from user input to outbound encrypted messages.

## What this revealed

The client-side code:

- watched keyboard activity;
- listened to field changes;
- packaged form state into message objects;
- encrypted those objects before sending them over the WebSocket.

### Static Key and IV Logic

By stopping execution close to the final encryption step, we found that the client used static key and IV values.

<CaptionImage
  src="/img/key_iv_1.png"
  alt="Key_IV"
  caption="Key & IV"
/>

:::note
| Key (ASCII) | IV (hex) | 
| --- | --- | 
| ZQMWLSPXJRDHKTNV | 59478a435546b3a1474fdb0c584c43d2 
:::

This is one of the biggest weaknesses in the kit.

If the browser can encrypt data, the browser must already contain what is needed to do that encryption. When those values are static and embedded in front-end code, they can be recovered.

That means the operator gets secrecy only against casual observers, not against anyone doing systematic reversing.

It also means defenders can:

- decode traffic once the scheme is understood;
- recover event structures;
- compare crypto reuse across variants;
- use the implementation as a fingerprint for related campaigns.

At that point, the objective was no longer just `can the encryption be reversed?` but `what exactly is the phishing panel collecting, and when?`

The decrypted stream showed that the site was sending data continuously, including partial values during typing.

That means the kit was not waiting for a final submit action. It was harvesting progressively.

Representative event types included:

- `submitData`
- `changleField`
- `enterCardNumber`
- `userSiteConfig`

The repeated typo `changleField` is especially interesting because stable typos often become strong kit fingerprints.

#### What the decrypted traffic showed

The backend received:

- first name;
- last name;
- email address;
- cardholder name;
- payment card number;
- expiry date;
- CVV;
- page routing changes;
- intermediate edits and backspaces.

# Security Analysis

1. `The campaign used scale as protection`

The many TLD variants show that the operator did not rely on stealth alone. They relied on infrastructure spread and replacement capacity.

2. `The page collected data before final submission`

This is one of the most important findings. The panel captured value while the victim typed, which reduced the operator’s dependence on full form completion.

3. `The encryption was operational, not robust`

The actor used encryption to conceal the workflow from simple inspection, but static client-side cryptographic material made the scheme recoverable.

4. `The kit appears reusable and template-based`

The routing model, repeated event names, encrypted config objects, and multilingual remnants all suggest a reusable phishing application rather than a one-off manually crafted lure.

5. `Small implementation details became strong fingerprints`

The `/ma/` route, the `changleField` typo, the WebSocket framing, and the crypto pattern are all useful artifacts for clustering and future detection.

# Conclusion

The `amendes-justice-gov.xxx` infrastructure appears to be a structured phishing operation built around scalable deployment, real-time collection, and a reusable client-side application.

The logic of the investigation moved from surface discovery to deeper protocol understanding:

- first, identify where the campaign lives;
- then, prove which infrastructure belongs together;
- next, inspect how the victim browser communicates;
- finally, reverse the client logic to reveal what is being collected.

The most important result is that the phishing page did not simply impersonate a payment site visually. It implemented a live collection workflow that transmitted victim data progressively over encrypted WebSocket messages.

That makes the infrastructure more informative for defenders, because once the client-side logic is reversed, the campaign exposes not only its domains, but also its internal workflow, event structure, and collection strategy.
